## 并发编程基础

[Github Repo Link](https://github.com/marco-hmc/cpp_rx)

### 概述

cpu 的单核性能提升接近瓶颈，多核是未来 cpu 的趋势。而并发技术就能充分利用多核特性。具体来说，并发编程是指在同时执行多个任务或线程，使得这些任务或线程可以在同一时间段内交替执行，从而提高程序的执行效率和资源利用率。

同时并发的意义也在于提升处理任务数量（请求个数）的能力，比如说即时通讯系统中，发起多个会话就是如此。网络请求的高并发往往就非常依赖于协程，十万并发的时候，服务器可开不了十万个线程，但是开十万个协程还是可以的。

在单机编程中，并发编程的主要手段有：
- **多线程**：共享内存空间的轻量级执行单元
- **多进程**：独立内存空间的重量级执行单元  
- **异步编程（协程）**：用户态管理的轻量级并发

### 1. 三种并发模式对比

#### 横向对比（进程 vs 线程 vs 协程）

- 调度机制
  - 进程：由内核调度；切换涉及完整地址空间（页表、FD、寄存器）。
  - 线程：由内核调度；共享进程地址空间，切换保存私有栈与寄存器。
  - 协程：用户态调度；由程序代码控制切换点（await/yield），操作系统不感知。

- 切换开销（量级）
  - 进程：约 1–10 微秒；TLB 刷新、内存映射维护，CPU 开销高。
  - 线程：约 0.1–1 微秒；需要系统调用/内核态中断。
  - 协程：约 10–100 纳秒；仅保存少量寄存器，无内核态切换。

- 内核介入与阻塞
  - 进程：是；系统调用/阻塞 I/O 直接阻塞该进程。
  - 线程：是；阻塞 I/O 阻塞该线程。
  - 协程：否（切换）；但若使用阻塞 I/O 会卡住线程，需配合非阻塞 I/O/事件循环。

- 资源/隔离
  - 进程：强隔离，天然避免共享内存数据竞争；崩溃不影响其他进程。
  - 线程：共享内存与句柄，需显式同步（锁/原子）；单线程崩溃影响整个进程。
  - 协程：共享所在线程上下文；单线程内高并发但需避免阻塞整个线程。

- 通信与同步方式
  - 进程：IPC（管道、消息队列、共享内存、套接字）；成本较高、需序列化。
  - 线程：共享内存 + 互斥量、条件变量、原子操作；成本低但有竞态风险。
  - 协程：事件循环 + 非阻塞 I/O；少用锁，更依赖任务调度与消息传递。

- 适用场景
  - 进程：高稳定性/隔离需求（崩溃隔离）、计算密集型、服务间隔离。
  - 线程：I/O 密集、需要并行利用多核；通用服务器/线程池。
  - 协程：海量连接/高并发 I/O（WebSocket、爬虫）；低延迟、单线程内高并发。

- 典型瓶颈
  - 进程：IPC 开销与内存复制、上下文切换抖动。
  - 线程：锁竞争、伪共享、过度线程数导致调度开销。
  - 协程：阻塞调用卡住事件循环；长耗时 CPU 计算阻塞整个线程。

- 语言/运行时限制
  - 进程：跨进程数据需序列化/复制，开发复杂度较高。
  - 线程：部分语言有 GIL（如 Python）限制多核并行；C/C++/Rust 不受此限。
  - 协程：需非阻塞 I/O 生态配合；调度正确性高度依赖框架。

- 选择建议
  - 高稳定隔离：优先多进程；跨故障域部署更安全。
  - 真并行利用多核：线程（或多进程 + 共享内存/IPC）；配合线程池。
  - 海量 I/O/低延迟：协程/异步框架；确保全链路非阻塞。
  - 简单且正确优先：先用线程池/成熟异步框架，再按瓶颈定位优化（无锁/协程/进程隔离）。

### 2. 进程和线程详解

#### 2.1 执行状态

进程和线程的状态转换：

```
    +-------+       +-------+
    | New   |  ->   | Ready |
    +-------+       +-------+
                      ^  |
                      |  v
    +-----------------------+
    |       Running         |
    +-----------------------+
        |  ^             |
        v  |             v
    +-------+        +-------+
    |Waiting|        |Terminated|
    |Blocked|        |          |
    +-------+        +-------+
```

**状态说明**：
- **New**：进程/线程正在被创建
- **Ready**：等待被分配 CPU 时间片
- **Running**：正在执行（唯一消耗 CPU 资源的状态）
- **Waiting/Blocked**：等待某个事件（I/O、锁等）
- **Terminated**：执行完成或被终止

**进入阻塞状态的情况**：
- 拿不到锁（`mtx.lock()`）
- 调用 `wait()` 等待条件变量
- 调用 `sleep()` 主动休眠
- 发起内存/硬盘/网络等 I/O 操作

#### 2.2 调度机制

**调度单位**：
- Linux 中调度的基本单位是线程（轻量级进程 LWP）
- 每个线程被视为独立的调度实体

**调度策略**：

1. **完全公平调度器（CFS）**：
   - Linux 默认调度器
   - 使用红黑树管理可运行任务
   - 根据虚拟运行时间进行调度

2. **实时调度器**：
   - `SCHED_FIFO`：先进先出，优先级抢占
   - `SCHED_RR`：轮转调度，固定时间片

3. **特殊调度器**：
   - `SCHED_BATCH`：批处理任务
   - `SCHED_IDLE`：最低优先级任务

**优先级设定**：
- **实时优先级**：1-99（数值越大优先级越高）
- **普通优先级**：-20 到 19（数值越小优先级越高）

#### 2.3 进程 vs 线程

| 特性 | 进程 | 线程 |
|------|------|------|
| **资源分配** | 独立的地址空间、内存、文件描述符 | 共享进程资源 |
| **创建开销** | 高（分配独立资源） | 低（共享资源） |
| **切换开销** | 高（完整上下文切换） | 低（轻量级切换） |
| **通信方式** | IPC（管道、消息队列、共享内存） | 共享内存 + 同步机制 |
| **隔离性** | 强隔离，互不影响 | 共享资源，需要同步 |
| **稳定性** | 高（进程崩溃不影响其他进程） | 低（线程崩溃影响整个进程） |

#### 2.4 线程资源共享

**私有资源**：
- **栈内存**：局部变量和函数调用上下文
- **寄存器**：程序计数器等处理器状态
- **线程局部存储（TLS）**：线程私有的全局变量副本

**共享资源**：
- **堆内存**：动态分配的内存空间
- **全局变量和静态变量**：进程级别的数据
- **文件和网络资源**：系统资源句柄

### 3. 性能开销分析

#### 3.1 切换开销对比

**进程切换开销**：
- 地址空间转换（切换页表、刷新 TLB）
- 硬件上下文切换（内核栈、寄存器）
- 保存/恢复进程状态（PC、虚拟内存信息）
- 用户态/内核态切换

**线程切换开销**：
- 硬件上下文切换（相对较小）
- 保存/恢复线程状态（PC、栈指针）
- 共享地址空间，CPU Cache 命中率较高

**协程切换开销**：
- 用户态切换，无系统调用
- 仅保存 PC、栈指针等少量数据
- CPU Cache 命中率最高

#### 3.2 性能瓶颈

**多进程瓶颈**：
- 进程间通信开销（序列化、内存复制）
- 上下文切换抖动（高频切换导致 CPU 利用率下降）

**多线程瓶颈**：
- 锁竞争（共享资源导致线程阻塞）
- 内核调度延迟（线程数超过 CPU 核心数）

**协程瓶颈**：
- CPU 计算阻塞（长耗时计算阻塞整个线程）
- 系统调用阻塞（同步 I/O 导致事件循环停滞）

### 4. 重要概念辨析

#### 4.1 可重入函数 vs 线程安全函数

**可重入函数**：
- 可以在任何时刻被中断并安全重新进入
- 只使用栈上数据，不使用静态/全局变量
- 不调用不可重入函数
- **无锁**（重要！）

**线程安全函数**：
- 可以被多个线程同时调用而不产生竞态条件
- 可以使用锁保护共享数据
- 可以使用静态/全局数据（有适当保护）

**关系**：
- 可重入函数是线程安全函数的真子集
- 可重入函数不能使用锁（信号处理需求）
- 线程安全函数可以通过加锁实现

#### 4.2 不可重入的情况

1. **使用静态数据结构**（如 `getpwnam`）
2. **调用 `malloc` 或 `free`**（修改堆链接表）
3. **使用标准 IO 函数**（如 `printf`，使用全局状态）
4. **调用 `longjmp` 或 `siglongjmp`**（可能导致数据部分更新）

#### 4.3 可重入函数的应用场景

**信号处理程序**：
- 处理异步信号的函数
- 可以在任何时候被调用
- 必须是可重入的以避免数据不一致

**中断处理程序**：
- 处理硬件中断
- 在嵌入式系统或内核中使用
- 必须保证系统稳定性

**特点**：
- 信号处理函数有极高优先级
- 不能使用锁（避免死锁）
- 被中断的线程必须等待信号处理完成

### 总结

并发编程需要根据具体场景选择合适的方案：

- **计算密集型**：考虑多进程或线程池
- **I/O 密集型**：优先考虑协程或异步编程
- **高稳定性要求**：选择多进程
- **低延迟要求**：选择协程
- **复杂同步需求**：谨慎使用多线程，注意锁的设计

理解这些基本概念是进行高效并发编程的基础。


- **进程生命周期管理**：创建 → 就绪 → 运行 → 阻塞 → 终止
- **调度算法体系**：
  - 时间片轮转：公平性保证，防止饥饿
  - 优先级调度：重要任务优先，响应敏感
  - 多级反馈：动态优先级调整，自适应负载
- **上下文切换机制**：保存/恢复执行状态
- **同步与通信**：防止竞态条件，协调进程协作

**进程状态转移与系统事件的对应**：

```
创建 → 就绪 ⇄ 运行 → 终止
        ↓     ↑
        阻塞 ←┘
         ↑
    系统调用/I/O等待
```
