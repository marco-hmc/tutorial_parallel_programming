## 进程那些事儿

### 1. concepts

#### 1.1 什么是进程

进程是**操作系统进行资源分配和调度的基本单位**，每个进程拥有独立的内存空间（代码段、数据段等），互不干扰；进程间通信需通过特殊机制（如管道、Socket）。

进程是操作系统进行资源分配和调度的基本单位,是应用程序在操作系统中的一次运行过程.进程拥有自己独立的内存空间,每个进程中都有至少一个线程.进程之间的通信需要使用特殊的进程间通信机制.

- 优点：
  - 进程是互相独立的，不涉及锁什么之类的，代码编写方便，调试方便
  - 进程是独立的，所以稳定性更高，一个进程挂了，其他也不会挂；
- 缺点：
  - 进程创建慢，占用内存多，进程的上下文切换代价高昂
  - 进程之间通信麻烦且慢，常用且好的多进程通信机制一般为 socket，所以通信内容一般为指令而非数据

##### 1.1.1 守护进程和后台进程

1. **守护进程**
   - **定义**：守护进程（Daemon 进程）是在后台长期运行的服务进程，独立于控制终端，没有交互界面。这类进程通常用于提供系统级服务，例如 Web 服务器 `nginx`、数据库服务 `mysql` 等。它们不受用户登录或注销的影响，一般在开机时自动启动，直到系统关闭才终止。
   - **实现思路**：标准的守护进程创建需借助 `fork` 函数配合一系列操作，从而确保进程能够脱离终端和父进程的控制。其核心步骤如下：
     - **第一次 fork**：父进程创建子进程后即刻退出，使得子进程成为“孤儿进程”，随后由 init 进程（或在现代系统中由 systemd 等初始化进程）接管，以此脱离原终端的控制。
     - **设置会话 ID（setsid）**：子进程调用 `setsid()` 函数创建新的会话，并成为该会话的组长，这一步能让进程彻底脱离原终端所属的进程组。
     - **第二次 fork（可选）**：再次创建子进程并退出当前进程，此操作可避免子进程成为会话组长，防止其意外获取终端。
     - **切换工作目录**：通常将工作目录切换到根目录（`chdir("/")`），这样做是为了避免占用可卸载的文件系统，确保进程运行不受文件系统挂载状态的影响。
     - **关闭文件描述符**：关闭从父进程继承而来的与终端相关的标准输入（0）、标准输出（1）和标准错误（2）等文件描述符，防止进程与原终端产生不必要的交互。
     - **重定向文件描述符**：将标准输出和标准错误重定向到 `/dev/null` 或日志文件，这样既能避免进程输出无意义的内容，又方便记录进程运行过程中的重要信息。
   - **Linux 系统中的查看方式**：在 Linux 系统中，守护进程通常以 `d` 结尾（如 `sshd`、`crond`）。可以通过以下命令查看：
     - `ps -ef | grep -v grep | grep -E 'daemon|d$'`：此命令用于筛选进程名称中包含 `daemon` 或结尾为 `d` 的进程，但这并非绝对的判断标准，仅作为参考。
     - `ps aux | grep '^root'`：由于多数守护进程以 root 身份运行，结合此条件可更精准地筛选守护进程。
     - `systemctl list-units --type=service`：此命令用于查看系统服务，因为多数系统服务对应的进程即为守护进程。
2. **后台进程**
   - **定义**：后台进程是指在终端中启动后，不会阻塞终端，用户能够在终端中继续执行其他命令的进程。通常通过在命令末尾加上 `&` 符号来启动后台进程。
   - **与守护进程的区别**：后台进程虽然也在后台运行，但严格意义上它与守护进程存在区别。后台进程一般仍与终端存在某种联系，例如其输出可能会显示在终端上，当终端关闭时，后台进程可能会受到影响（除非采取特殊处理）。而守护进程完全脱离终端，独立运行，不受终端状态的影响。在功能和应用场景方面，后台进程常用于执行一些相对独立且不需要用户实时关注的短期任务，例如在终端中启动一个数据处理脚本，用户在脚本执行过程中仍可使用终端进行其他操作。守护进程则更多用于提供持续的系统级服务，为整个系统或网络中的其他组件提供支持，如 Web 服务器守护进程持续监听网络请求，为客户端提供网页服务。
   简单来说，守护进程是 “系统的后台服务”，而后台进程是 “用户的后台任务”。

##### 1.1.2 孤儿进程和僵尸进程

1.  孤儿进程

    - **定义**：当一个父进程退出，而它的一个或多个子进程仍在运行时，这些子进程就成为孤儿进程。孤儿进程会被 `init` 进程（进程号为 1）收养，由 `init` 进程负责完成它们的状态收集工作。
    - **特点**：孤儿进程不会对系统造成危害，因为 `init` 进程会妥善处理它们的状态收集与资源释放。

2.  僵尸进程
    - **定义**：一个进程使用 `fork` 创建子进程后，如果子进程退出，而父进程没有调用 `wait` 或 `waitpid` 获取子进程的状态信息，子进程的进程描述符仍会保存在系统中，这类进程即为僵尸进程。在 Linux 系统中，僵尸进程是已退出但资源未被回收的进程。子进程退出时，内核会保留其 PID、退出状态等信息，等待父进程通过 `wait()` 或 `waitpid()` 回收；若父进程未调用这些函数，子进程会成为僵尸进程，其状态标记为 `Z`（Zombie）。
    - **产生原因**
      - 父进程未调用 `wait`/`waitpid` 系统调用回收子进程资源。
      - 父进程存在逻辑缺陷，例如忽略子进程退出信号 `SIGCHLD`，或者长期运行但未处理子进程状态。
    - **查看方法（Linux 系统）**
      - 使用 `ps` 命令查看状态为 `Z` 的进程：
      ```bash
      ps -ef | grep Z  # 状态列显示 Z 的即为僵尸进程
      # 或更直观的格式：
      ps aux | awk '$8=="Z"'  # 筛选状态（第8列）为 Z 的进程
      ```
    - **特点**
      - 任何一个子进程（`init` 除外）在 `exit()` 之后，不会立即消失，而是留下一个称为僵尸进程（Zombie）的数据结构，等待父进程处理。
      - 僵尸进程的进程号会一直被占用，而系统所能使用的进程号有限。若大量产生僵尸进程，会因进程号耗尽导致系统无法产生新进程。
    - **危害**：僵尸进程占用系统的进程号资源，若不及时处理，可能致使系统无法创建新进程。
    - **解决办法**
      - **父进程主动回收**：父进程通过 `wait()` 或 `waitpid()` 系统调用获取子进程退出状态，内核随即释放僵尸进程的资源。示例代码如下：
    ```c
    #include <sys/wait.h>
    int status;
    wait(&status);  // 阻塞等待任意子进程退出并回收
    ```
        - **父进程处理 `SIGCHLD` 信号**：子进程退出时会向父进程发送 `SIGCHLD` 信号，父进程可注册信号处理函数，在信号触发时调用 `wait` 回收。示例代码如下：
    ```c
    #include <signal.h>
    void handle_sigchld(int sig) {
        wait(NULL);  // 收到信号后回收子进程
    }
    signal(SIGCHLD, handle_sigchld);  // 注册信号处理
    ```
        - **终止父进程**：若父进程无法修复，终止父进程后，僵尸进程会被 `init` 进程（或 `systemd`）接管，`init` 会自动调用 `wait` 回收其资源。

- 僵尸进程的危害场景
  1. **场景描述**：假设有一个进程定期产生子进程，这些子进程执行少量任务后即退出，生命周期很短。然而，父进程只管生成新子进程，对子进程退出后的情况不管不顾。一段时间后，系统中就会出现大量僵尸进程，使用 `ps` 命令查看，会看到许多状态为 `Z` 的进程。
  2. **根本原因**：严格来讲，僵尸进程并非问题根源，产生大量僵尸进程的父进程才是罪魁祸首。
  3. **处理方法**：终止产生大量僵尸进程的父进程（通过 `kill` 发送 `SIGTERM` 或者 `SIGKILL` 信号）。之后，这些僵尸进程会变成孤儿进程，被 `init` 进程接管。`init` 进程会调用 `wait()` 释放它们占用系统进程表中的资源，从而清理掉这些僵死的孤儿进程。

#### 1.2 进程 ID

进程通过以下 ID 实现系统级管理和区分：

| 标识     | 全称      | 作用                                                                        |
| -------- | --------- | --------------------------------------------------------------------------- |
| **PID**  | 进程 ID   | 唯一标识一个进程（如 `ps` 命令显示的第一列数字）。                          |
| **PGID** | 进程组 ID | 多个相关进程组成进程组（如管道中的多个命令），PGID 通常等于组长进程的 PID。 |
| **SID**  | 会话 ID   | 一个会话包含多个进程组（如终端启动的所有进程），用于管理终端相关进程。      |

**示例**：在终端运行 `ls | grep txt` 时，`ls` 和 `grep` 属于同一进程组（共享 PGID），且同属终端启动的会话（共享 SID）。

#### 1.3 进程的组成

一个进程由 4 部分核心组件构成，它们对于进程的运行和管理缺一不可：

1. **进程控制块（PCB，Process Control Block）** PCB 如同进程的“身份证”，是内核维护的结构体，记录着进程所有关键信息，也是进程存在的唯一标志（当 PCB 被释放时，进程即消亡）。不同操作系统中 PCB 的具体实现可能略有不同，其通常包含以下几类信息： 

  - **进程描述信息**：进程标识符（PID）、进程组标识符（PGID）、会话标识符（SID）以及父进程标识符（PPID）等，用于唯一标识进程及其所属关系。 
  - **进程状态信息**：包括运行态（R）、就绪态（S）、阻塞态（D）、僵尸态（Z）等进程状态，还有进程优先级、未决信号等信息，这些决定了进程在系统中的调度和执行顺序。 
  - **资源分配清单**：进程打开的文件描述符（如标准输入 `stdin`、标准输出 `stdout` 等）、内存页表（负责将虚拟地址映射到物理地址，实现进程对内存资源的管理）以及信号量等，记录了进程运行过程中所占用的各类资源。 
  - **CPU 上下文**：包含寄存器（如程序计数器 PC，用于指示下一条要执行的指令地址；栈指针 SP，指向当前进程栈的栈顶）、状态寄存器（记录 CPU 当前的状态）等信息。这些信息在进程切换时发挥关键作用，能够确保进程在重新执行时恢复到之前的运行状态。

2. **数据段**：进程运行过程中使用的各种数据，例如程序中定义的变量等，都存储在此段。它是进程处理和操作数据的存储区域。
3. **程序段**：即程序的代码，是由一系列指令序列构成的，规定了进程要执行的具体操作和任务逻辑。
4. **堆栈段（文本未提及，但通常是进程组成部分）**：用于存储函数调用的参数、局部变量以及返回地址等。栈随着函数调用和返回进行动态的生长和收缩，为进程的函数调用和执行提供必要的环境支持。

进程控制块是操作系统对进程进行管理的重要依据，几乎所有与进程管理相关的操作都依赖于 PCB 中的内容。例如，进程调度程序依据 PCB 中的进程状态和优先级信息来决定哪个进程获得 CPU 资源；内存管理模块通过 PCB 中的内存页表信息来管理进程的内存使用等。

#### 1.4 进程上下文切换

当 CPU 从进程 A 切换到进程 B 时，需通过“上下文切换”来保存 A 的状态并恢复 B 的状态，具体步骤如下：

1. **保存进程 A 的上下文**：进程 A 的运行环境信息即进程的上下文（Context），其中包含 CPU 上下文（如寄存器、程序计数器 PC、栈指针 SP 等）。此时，将进程 A 的 CPU 上下文存入其进程控制块 PCB 中，这相当于为进程 A 的当前运行状态进行“存档”。
2. **更新进程 A 的状态并移至对应队列**：根据进程切换的原因，比如若因为时间片用完，那么将进程 A 的状态更新后，把其 PCB 移入就绪队列；若因等待 I/O 操作，则移入阻塞队列。
3. **选择下一个运行进程 B**：调度器依据特定的算法，例如优先级算法或者时间片轮转算法等，从就绪队列中挑选出下一个要运行的进程 B。
4. **恢复进程 B 的上下文并执行**：从进程 B 的 PCB 中加载其 CPU 上下文，恢复寄存器、PC 等信息，同时将进程 B 的状态更新为“运行态”，至此 CPU 开始执行进程 B 的指令。

**示例**：当进程 A 运行到 `sleep(1)` 时主动阻塞，此时就会触发上述 CPU 从进程 A 到进程 B 的切换流程。1 秒后进程 A 被唤醒，重新进入就绪队列，等待再次被调度。当再次被调度时，进程 A 将从 `sleep` 之后的指令继续恢复执行。

#### 1.5 进程调度

1. **先来先服务（FCFS，First - Come - First - Serve）**：按照作业到达任务队列的顺序进行调度。该算法是非抢占式的，实现起来较为简单。然而，它的效率不高，性能欠佳，相对有利于长作业（CPU 繁忙型作业），但对短作业（I/O 繁忙型作业）不利。因为短作业可能需要等待前面的长作业执行完毕，导致等待时间过长。
2. **短作业优先（SJF，Short - Job - First）**：每次从队列中选择预计运行时间最短的作业来运行。SJF 同样是非抢占式的，它优先照顾短作业，在性能方面表现较好，能够有效降低平均等待时间，提高系统吞吐量。不过，该算法对长作业不太友好，长作业可能会长时间处于等待状态，出现饥饿现象。并且，它完全没有考虑作业的优先紧迫程度，因此不适用于实时系统。
3. **最短剩余时间优先（SRTF，Shortest Remaining Time First）**：该算法在调度时，首先会挑选服务时间最短的作业运行。在当前作业运行期间，若有新作业到达系统，系统会比较新作业的服务时间与当前运行作业的剩余服务时间。若新作业的服务时间更短，则发生抢占，即暂停当前作业，调度新作业运行；否则，当前作业继续运行。SRTF 算法确保一旦有新的短作业或短进程进入系统，能够迅速得到处理，减少了短作业的等待时间。
4. **高响应比优先调度算法（HRRF，Highest Response Ratio First）**：这是一种非抢占式的调度算法，主要应用于作业调度。其基本思想是，每次进行作业调度时，先计算后备作业队列中每个作业的响应比，然后挑选响应比最高的作业投入系统运行。响应比的计算公式为：响应比 = (等待时间 + 服务时间) / 服务时间 = 等待时间 / 服务时间 + 1。由于每次调度都需要计算所有作业的响应比，所以该算法相对比较耗费系统资源。
5. **时间片轮转**：常用于分时系统的进程调度。其基本思想是，系统将 CPU 的处理时间划分为若干个固定长度的时间片（$q$）。进程按照到达的先后顺序排列成队列，每次调度选择队首的进程执行。当该进程执行完 1 个时间片 $q$ 后，计时器会发出时钟中断请求，此时该进程被移至队尾。之后每次调度都重复此过程。通过这种方式，该算法能够在给定的时间内响应所有用户的请求，从而达到分时系统的目的，保证每个用户的操作都能得到及时反馈。
6. **多级反馈队列（Multilevel Feedback Queue）**：该算法设有多个不同优先级的队列，每个队列有不同的时间片长度。新进程进入系统后，首先被放入最高优先级队列。在该队列中，进程按照时间片轮转方式执行。若在一个时间片内进程未执行完，则被移至下一级队列。当下一级队列的所有进程都执行过一轮（一个时间片）后，上一级队列中未执行完的进程才有机会再次执行。优先级越高的队列，时间片越短。这种算法结合了时间片轮转和优先级调度的优点，既能保证短作业尽快完成，又能兼顾长作业，避免其长时间得不到执行。同时，对于 I/O 繁忙型进程也比较友好，因为它们通常能在高优先级队列中快速完成 I/O 操作，减少等待时间。

#### 1.6 进程切换发生的原因

进程切换（CPU 从一个进程切换到另一个进程）的触发条件：

1. **主动放弃 CPU**：
   - 进程执行 `sleep`、`wait` 等阻塞操作（等待 I/O、信号等），进入阻塞状态。
   - 进程调用 `yield` 主动让出 CPU（仅建议，调度器可忽略）。
2. **被动剥夺 CPU**：
   - 时间片耗尽：分时系统中，进程用完分配的 CPU 时间片，被调度器强制切换。
   - 优先级抢占：高优先级进程就绪时，抢占低优先级进程的 CPU。
   - 硬件中断：如 I/O 完成中断，内核处理中断后可能触发调度（如唤醒高优先级进程）。

#### 1.7 处理进程切换的步骤

进程切换由内核的调度器完成，核心是保存当前进程状态、恢复目标进程状态，步骤：

1. **保存当前进程上下文**：  
   将当前进程的 CPU 寄存器（如程序计数器 PC、栈指针 SP、通用寄存器）、进程状态（如运行态 → 就绪态）保存到其 PCB（进程控制块）中。
2. **更新进程 PCB 信息**：  
   修改当前进程的状态（如从“运行”改为“就绪”或“阻塞”），更新在就绪队列/阻塞队列中的位置。
3. **选择下一个运行进程**：  
   调度器根据调度算法（如优先级调度、时间片轮转）从就绪队列中选择目标进程。
4. **恢复目标进程上下文**：  
   从目标进程的 PCB 中加载 CPU 寄存器数据，恢复程序计数器、栈指针等，使其从上次暂停的位置继续执行。
5. **切换地址空间**：  
   更新内存管理单元（MMU）的页表寄存器，映射到目标进程的地址空间，确保内存访问正确。

### 2. 进程创建

#### 2.1 fork，vfork，clone

进程创建是操作系统中的重要操作，通常通过 `fork`（创建子进程） + `exec`（加载新程序）来实现。以下详细介绍进程创建的核心步骤以及 `fork`、`vfork`、`clone` 这三个函数在进程创建中的作用。

**进程创建核心步骤**：

1. **复制进程资源（fork）**：  
   父进程调用 `fork()` 函数，此时内核会为子进程复制父进程的多项资源，包括地址空间（代码段、数据段、堆栈段）、文件描述符以及信号处理等。在现代操作系统中，为了提高效率，广泛采用“写时复制（Copy - On - Write, COW）”技术进行优化。这意味着在子进程创建初期，父子进程共享这些资源，仅当其中一方对资源进行修改时，系统才会真正为修改的部分进行复制，从而减少了不必要的资源复制开销。
2. **分配进程标识（PID）**：  
   内核为新创建的子进程分配一个唯一的进程标识符（PID）。同时，将子进程的状态设置为“就绪”，表明该子进程已准备好，可以参与 CPU 的调度竞争。
3. **加载新程序（可选，exec）**：  
   如果子进程需要运行一个全新的程序，就会调用 `exec` 系列函数，例如 `execl`、`execv`、`execle`、`execve` 等。这些函数的作用是用新程序的代码和数据替换子进程当前的地址空间，从而使得子进程开始执行新的程序逻辑。不过，在这个过程中，子进程的 PID 等核心属性会保持不变。以 `execl` 函数为例，其函数原型为 `int execl(const char *path, const char *arg, ...);`，其中 `path` 是要执行的程序的路径，`arg` 是传递给新程序的参数列表，并且参数列表以 `NULL` 结尾。例如，要执行 `/bin/ls` 命令并列出当前目录下的文件，可以使用 `execl("/bin/ls", "ls", "-l", NULL);`。
4. **加入调度队列**：  
   完成上述步骤后，子进程进入就绪队列，等待 CPU 调度执行。调度器会根据特定的调度算法（如先来先服务、时间片轮转等）决定何时让该子进程获得 CPU 资源，进而开始执行其程序代码。

**`vfork` 函数**：`vfork` 函数也是用于创建子进程，但它与 `fork` 有所不同。`vfork` 创建的子进程与父进程共享地址空间，这意味着子进程对数据的修改会直接反映在父进程中。而且，在子进程调用 `exec` 或 `exit` 之前，父进程会被阻塞，以确保子进程先执行关键操作，避免数据竞争等问题。这种特性使得 `vfork` 在一些特定场景下（如子进程紧接着要执行 `exec` 加载新程序的情况）能够提高效率，减少资源开销。

**`clone` 函数**：`clone` 函数提供了更为灵活的进程创建方式。它允许用户精细控制子进程对父进程资源的继承情况。通过传递不同的参数，用户可以指定子进程是否与父进程共享地址空间、文件描述符、信号处理等资源。例如，可以通过设置参数让子进程只共享部分资源，从而在创建进程时实现更细粒度的资源管理。`clone` 函数的原型为 `int clone(int (*fn)(void *), void *child_stack, int flags, void *arg, ...);`，其中 `fn` 是子进程开始执行的函数，`child_stack` 是为子进程分配的栈空间，`flags` 用于指定资源共享等标志位，`arg` 是传递给 `fn` 函数的参数。这种灵活性使得 `clone` 函数适用于一些对进程创建有特殊需求的场景，如实现线程库等。

#### 3.1 管道和FIFO

* **管道**
- 管道:管道是半双工的,双方需要通信的时候,需要建立两个管道.管道的实质是一个内核缓冲区,进程以先进先出的方式从缓冲区存取数据:管道一端的进程顺序地将进程数据写入缓冲区,另一端的进程则顺序地读取数据,该缓冲区可以看做一个循环队列,读和写的位置都是自动增加的,一个数据只能被读一次,读出以后再缓冲区都不复存在了.当缓冲区读空或者写满时,有一定的规则控制相应的读进程或写进程是否进入等待队列,当空的缓冲区有新数据写入或慢的缓冲区有数据读出时,就唤醒等待队列中的进程继续读写.管道是最容易实现的匿名管道 pipe 和命名管道除了建立,打开,删除的方式不同外,其余都是一样的.匿名管道只允许有亲缘关系的进程之间通信,也就是父子进程之间的通信,命名管道允许具有非亲缘关系的进程间通信.

  管道的底层实现 https://segmentfault.com/a/1190000009528245

- 通常指无名管道，是 UNIX 系统 IPC 最古老的形式

- 它是**半双工**的（即数据只能在一个方向上流动），具有固定的读端和写端
- 它只能用于**具有亲缘关系的进程之间的通信**（也是父子进程或者兄弟进程之间）
- 它可以看成是一种特殊的文件，对于它的读写也可以使用普通的 read、write 等函数。但是它不是普通的文件，并**不属于其他任何文件系统**，并且**只存在于内存**中
- 原型

```c
1 #include <unistd.h>
2 int pipe(int fd[2]);    // 返回值：若成功返回0，失败返回-1
```

管道是通过调用 pipe 函数创建的，fd[0] 用于读，fd[1] 用于写。

```c
#include <unistd.h>
int pipe(int fd[2]);
```

它具有以下限制：

- 只支持半双工通信（单向交替传输）；
- 只能在父子进程或者兄弟进程中使用。

* **FIFO**

- 也称为命名管道，它是一种文件类型

- FIFO 可以在无关的进程之间交换数据，与无名管道不同
- FIFO 有路径名与之相关联，它以一种**特殊设备文件形式存在于文件系统**中
- FIFO 的通信方式类似于在进程中使用文件来传输数据，只不过 FIFO 类型文件同时具有管道的特性。**在数据读出时，FIFO 管道中同时清除数据，并且“先进先出”**
- 原型

```c
1 #include <sys/stat.h>
2 // 返回值：成功返回0，出错返回-1
3 int mkfifo(const char *pathname, mode_t mode);
```

- 其中的`mode`参数与`open`函数中的 `mode `相同。一旦创建了一个 FIFO，就可以用一般的文件 I/O 函数操作它

  - 当 open 一个 FIFO 时，是否设置非阻塞标志（`O_NONBLOCK`）的区别：
    - 若没有指定`O_NONBLOCK`（默认），只读 open 要阻塞到某个其他进程为写而打开此 FIFO。类似的，只写 `open` 要阻塞到某个其他进程为读而打开它
    - 若指定了`O_NONBLOCK`，则只读`open`立即返回。而只写`open`将出错返回 -1 如果没有进程已经为读而打开该 FIFO，其`errno`置`ENXIO`。

也称为命名管道，去除了管道只能在父子进程中使用的限制。

```c
#include <sys/stat.h>
int mkfifo(const char *path, mode_t mode);
int mkfifoat(int fd, const char *path, mode_t mode);
```

FIFO 常用于客户-服务器应用程序中，FIFO 用作汇聚点，在客户进程和服务器进程之间传递数据。

#### 3.2 消息队列

消息队列就是一个消息的链表,是一系列保存在内核中消息的列表.用户进程可以向消息队列添加消息,也可以向消息队列读取消息. 消息队列与管道通信相比,其优势是对每个消息指定特定的消息类型,接收的时候不需要按照队列次序,而是可以根据自定义条件接收特定类型的消息. 可以把消息看做一个记录,具有特定的格式以及特定的优先级.对消息队列有写权限的进程可以向消息队列中按照一定的规则添加新消息,对消息队列有读权限的进程可以从消息队列中读取消息.

- **是消息的链接表，存放在内核中**。一个消息队列由一个标识符（即队列 ID）来标识

- 消息队列是面向记录的，其中的消息具有特定的格式以及特定的优先级
  - 消息队列独立于发送与接收进程。**进程终止时，消息队列及其内容并不会被删除**
  - **消息队列可以实现消息的随机查询，消息不一定要以先进先出的次序读取,也可以按消息的类型读取**
  - 原型

```c
#include <sys/msg.h>
// 创建或打开消息队列：成功返回队列ID，失败返回-1
int msgget(key_t key, int fslag);
// 添加消息：成功返回0，失败返回-1
int msgsnd(int msqid, const void *ptr, size_t size, int flag);
// 读取消息：成功返回消息数据的长度，失败返回-1
int msgrcv(int msqid, void *ptr, size_t size, long type,int flag);
// 控制消息队列：成功返回0，失败返回-1
int msgctl(int msqid, int cmd, struct msqid_ds *buf);
```

- 在以下两种情况下，`msgget`将创建一个新的消息队列：

  - 如果没有与键值 key 相对应的消息队列，并且 flag 中包含了`IPC_CREAT`标志位。
  - key 参数为`IPC_PRIVATE`。

  * 函数`msgrcv`在读取消息队列时，`type`参数有下面几种情况：
    - `type == 0`，返回队列中的第一个消息
    - `type > 0`，返回队列中消息类型为 `type` 的第一个消息
    - `type < 0`，返回队列中消息类型值小于或等于`type`绝对值的消息，如果有多个，则取类型值最小的消息
  * 可以看出，type 值非 0 时用于以非先进先出次序读消息。也可以把` type` 看做优先级的权值

相比于 FIFO，消息队列具有以下优点：

- 消息队列可以独立于读写进程存在，从而避免了 FIFO 中同步管道的打开和关闭时可能产生的困难；
- 避免了 FIFO 的同步阻塞问题，不需要进程自己提供同步方法；
- 读进程可以根据消息类型有选择地接收消息，而不像 FIFO 那样只能默认地接收。
- 匿名管道是跟随进程的,消息队列是跟随内核的,也就是说进程结束之后,匿名管道就死了,但是消息队列还会存在(除非显示调用函数销毁).
- 管道是文件,存放在磁盘上,访问速度慢,消息队列是数据结构,存放在内存,访问速度快.
- 管道是数据流式存取,消息队列是数据块式存取.

```c
#include <sys/types.h>
#include <sys/ipc.h>
#include <sys/msg.h>
int msgget(key_t key, int msgflg);

```

#### 3.3 信号(signal)

信号（Signal）是 Linux 系统中一种**用于进程间通信或异步事件通知的机制**，它本质上是在软件层次上对中断机制的模拟，允许内核或其他进程**异步地通知某个进程某个事件的发生**。

* 信号是 Linux 中一种重要的、**异步的进程间通信机制**；
* 支持硬件触发、软件调用及异常生成；
* 提供三种处理机制（默认、捕捉、忽略）；
* 某些信号不可被阻塞或忽略（如 `SIGKILL`、`SIGSTOP`）；
* 信号处理涉及**进程状态切换、堆栈备份与恢复、PC 寄存器重定向**等核心机制，是操作系统底层调度的重要组成部分。

---

* **信号的基本特性**

    * 信号可以在**任何时间发送给进程**，而不考虑进程当前的状态；
    * 如果进程**未处于执行状态**，信号将被**内核暂存**，直到进程恢复运行时再传递；
    * 如果进程**阻塞了某个信号**，该信号的传递将被延迟，直到解除阻塞；
    * 信号的通信开销较小，但**传递机制较复杂**；
    * 常通过 `<signal.h>` 提供的 API 进行信号处理；

---

* **信号的来源**

    信号可以由以下几类事件触发：
    1. **硬件来源**：
    * 例如用户按下 `Ctrl+C`，终端驱动会触发 `SIGINT` 信号。

    2. **软件来源**：
    * 调用系统函数如 `kill()`、`raise()`、`abort()`、`sigqueue()`、`setitimer()` 等；
        * `kill(pid, sig)`：向指定进程或进程组发送信号；
        * `raise(sig)`：向当前进程发送信号；
        * `sigqueue(pid, sig, value)`：向进程发送附带值的信号；
        * `setitimer()` / `alarm()`：定时触发信号（如 `SIGALRM`）；
        * `abort()`：立即终止进程，产生 `SIGABRT`；
    * 程序异常，如非法内存访问、除以 0、断点等；
    * **除零操作**；
    * **非法内存访问**（如访问空指针）；
    * **栈溢出**；
    * 某些异常还可能导致 core dump 文件生成，用于调试；

    ⚠️ `SIGKILL` 和 `SIGINT` 不会触发 core dump。

    3. **命令行工具**：
    * 使用 `kill` 命令可以向指定进程发送信号；
    * 键盘组合键如 `Ctrl+Z` 发送 `SIGTSTP`，`Ctrl+\` 发送 `SIGQUIT`。

* **进程对信号的响应方式**

进程接收到信号后，内核提供三种处理方式：

1. **执行默认操作**（多数信号都有默认处理行为）；
2. **捕捉信号**：注册自定义的信号处理函数；
3. **忽略信号**：让信号对进程无影响；

⚠️ 注意：两个信号不能被捕捉、忽略或阻塞：
* `SIGKILL (9)`：强制终止进程；
* `SIGSTOP (19)`：强制暂停进程；

---

* **信号的处理流程**

1. 信号进入内核，写入未决信号集；
2. 进程陷入内核态；
3. 内核检测未决信号；
4. 根据是否注册处理函数决定处理方式；
5. 信号处理完成后恢复原执行流。

1. **信号接收**

    * 信号由内核接收后，**放入进程 PCB 的未决信号集**；
    * 同时，**向进程发出中断请求**，使其从用户态陷入内核态；

    💡 此时，信号还未被处理，进程对其尚无感知。

2. **信号检测**

    信号检测发生在以下两个场景：

    * 进程**从内核态返回用户态**之前；
    * 进程**在内核态中从睡眠状态被唤醒**时；

    若检测到未决信号，则进入下一阶段。

3. **信号处理**

* 若未注册处理函数：使用内核默认方式；
* 若注册了处理函数：

  * 内核将当前**内核栈内容备份到用户栈**；
  * \*\*修改程序计数器（PC）\*\*指向用户注册的处理函数；
  * **回到用户态执行信号处理函数**；
  * 信号处理函数执行完毕后：

    * 再次进入内核；
    * 检查是否还有其他待处理信号；
    * 若没有，则**恢复栈和 PC**，继续原进程执行；

4. **信号处理顺序**

若多个信号到达，按上述流程依次处理所有信号。

---

* **信号处理时机**
    信号是**延迟处理**的：

  * 即使信号已经到达，进程不会立刻响应；
  * 只有当进程在**从内核态返回用户态前**或**从睡眠中被唤醒时**，才会检测并处理信号。

---

* **常见信号表（部分）**

| 信号名     | 编号 | 含义           |
| ------- | -- | ------------ |
| SIGHUP  | 1  | 终端挂断         |
| SIGINT  | 2  | 中断（如 Ctrl+C） |
| SIGKILL | 9  | 强制终止，无法捕捉    |
| SIGTERM | 15 | 请求程序正常终止     |
| SIGSTOP | 19 | 强制暂停，无法捕捉    |
| SIGSEGV | 11 | 无效内存访问（段错误）  |
| SIGALRM | 14 | 定时器超时        |
| SIGUSR1 | 10 | 用户定义信号1      |

#### 3.4 信号量

信号量是一个计数器，主要用于实现进程间的互斥与同步，而非用于存储进程间通信数据。与其他进程间通信（IPC）结构不同，它基于操作系统的 PV 操作，程序对其操作均为原子操作。信号量只有等待（P 操作）和发送（V 操作）两种操作，等待（P(sv)）会将其值减一或挂起进程，发送（V(sv)）则会将其值加一或恢复进程运行。信号量可用于进程间同步，若要在进程间传递数据，通常需结合共享内存。

信号量不仅支持对其值进行加 1 或减 1 的操作，还能加减任意正整数，并且支持信号量组。其相关函数原型如下：
```c
#include <sys/sem.h>
// 创建或获取一个信号量组：若成功返回信号量集ID，失败返回-1
int semget(key_t key, int num_sems, int sem_flags);
// 对信号量组进行操作，改变信号量的值：成功返回0，失败返回-1
int semop(int semid, struct sembuf semoparray[], size_t numops);
// 控制信号量的相关信息
int semctl(int semid, int sem_num, int cmd, ...);
```
当使用`semget`创建新的信号量集合时，必须指定集合中信号量的个数（即`num_sems`），通常设为 1；若引用现有的集合，则将`num_sems`指定为 0。

在`semop`函数中，`sembuf`结构定义如下：
```c
struct sembuf {
    short sem_num; // 信号量组中对应的序号，范围是0～sem_nums - 1
    short sem_op;  // 信号量值在一次操作中的改变量
    short sem_flg; // 可取值为IPC_NOWAIT, SEM_UNDO
};
```
其中，`sem_op`决定了一次操作中信号量的改变情况：
- **`sem_op > 0`**：表示进程释放相应的资源数，此时将`sem_op`的值加到信号量的值上。若有进程正在休眠等待此信号量，则唤醒它们。
- **`sem_op < 0`**：表示请求`sem_op`绝对值的资源。
    - **资源满足请求**：若相应的资源数足够，将该信号量的值减去`sem_op`的绝对值，函数成功返回。
    - **资源不满足请求**：此操作行为与`sem_flg`有关。
        - **`sem_flg`指定`IPC_NOWAIT`**：`semop`函数出错返回`EAGAIN`。
        - **`sem_flg`未指定`IPC_NOWAIT`**：将该信号量的`semncnt`值加 1，然后进程挂起，直到出现以下情况：
            - **资源满足请求**：相应资源数满足请求时，此信号量的`semncnt`值减 1，该信号量的值减去`sem_op`的绝对值，成功返回。
            - **信号量被删除**：函数`smeop`出错返回`EIDRM`。
            - **进程捕捉到信号**：进程捕捉到信号并从信号处理函数返回，此情况下将此信号量的`semncnt`值减 1，函数`semop`出错返回`EINTR`。
- **`sem_op == 0`**：进程阻塞直到信号量的相应值为 0。
    - **信号量已为 0**：函数立即返回。
    - **信号量值不为 0**：依据`sem_flg`决定函数动作。
        - **`sem_flg`指定`IPC_NOWAIT`**：出错返回`EAGAIN`。
        - **`sem_flg`未指定`IPC_NOWAIT`**：将该信号量的`semncnt`值加 1，然后进程挂起，直到出现以下情况：
            - **信号量值为 0**：将信号量的`semzcnt`的值减 1，函数`semop`成功返回。
            - **信号量被删除**：函数`smeop`出错返回`EIDRM`。
            - **进程捕捉到信号**：进程捕捉到信号并从信号处理函数返回，在此情况将此信号量的`semncnt`值减 1，函数`semop`出错返回`EINTR`。

在`semctl`函数中有多种命令，以下介绍两个常用的：
- **`SETVAL`**：用于初始化信号量为一个已知的值。所需的值作为联合`semun`的`val`成员来传递。在首次使用信号量之前，需要使用此命令设置信号量。（注：`semun`是一个用户自定义的联合类型，通常定义如下：`union semun { int val; struct semid_ds *buf; unsigned short *array; struct seminfo *__buf; };`，不同系统下定义可能略有差异）
- **`IPC_RMID`**：用于删除一个信号量集合。若不删除信号量，它将继续存在于系统中。即使程序退出，下次运行程序时可能引发问题，因为信号量是一种有限的资源。 

#### 3.5 共享内存

共享内存是一种进程间通信（IPC）方式，它允许多个进程共享一个给定的存储区，使这些进程能够直接对同一块内存区域进行读写操作，从而实现高效的数据共享与通信。

1. **共享内存的特点**

   - **高效性**：由于数据无需在进程之间复制，共享内存是最快的 IPC 方式之一。进程可直接对共享内存进行存取，大大提高了数据传输效率。
   - **同步需求**：然而，共享内存本身未提供同步机制。多个进程同时操作共享内存可能导致数据不一致问题，因此通常需要结合锁、信号量等机制来同步对共享内存的访问。

2. **共享内存的实现原理** 共享内存通过将同一块内存分别映射到不同进程的地址空间来实现进程间通信。这样，多个进程就能够访问和修改同一块内存区域的数据。但由于缺乏内置的互斥与同步机制，当多个进程同时对共享内存进行读写操作时，可能破坏内存中的数据，所以在实际应用中，同步与互斥机制需要由用户自行实现。

3. **共享内存的相关限制** 在 `/proc/sys/kernel/` 目录下，记录着共享内存的一些限制参数，如一个共享内存区的最大字节数 `shmmax`，系统范围内最大共享内存区标识符数 `shmmni` 等。虽然可以手动调整这些参数，但不推荐随意操作，以免影响系统稳定性。

#### 3.6 套接字

与其它通信机制不同的是，它可用于不同机器间的进程通信。

- 套接字:套接口也是一种进程间通信机制,与其他通信机制不同的是,它可用于不同设备及其间的进程通信.
