## 进程那些事儿

### 1. concepts

#### 1.1 什么是进程

进程是操作系统进行资源分配和调度的基本单位，也是应用程序在操作系统中的一次运行过程。每个进程都拥有独立的内存空间，包括代码段、数据段等，各进程之间的内存空间相互隔离、互不干扰。正因如此，进程间通信需要借助特殊机制，例如管道、Socket 等。

每个进程中至少包含一个线程。线程是进程中的实际执行单元，同一进程内的线程共享该进程的资源。

- 进程的优点
    * 代码编写与调试便捷**：由于进程相互独立，在编写代码时无需考虑诸如锁机制等复杂的同步问题，这使得代码编写更为简便，调试也相对容易。
    * 稳定性高**：进程之间彼此独立，一个进程出现故障（如挂掉），不会影响其他进程的正常运行，从而提高了整个系统的稳定性。

- 进程的缺点
    * 资源开销大**：进程创建过程相对复杂且耗时，同时占用较多内存资源。此外，进程的上下文切换需要保存和恢复大量的状态信息，代价高昂，会在一定程度上影响系统性能。
    * 通信不便且效率低**：进程之间的通信较为麻烦，速度相对较慢。常用且效果较好的多进程通信机制一般是 socket，但这种方式更适合传递指令，对于大量数据的传输并不高效。 

##### 1.1.1 守护进程和后台进程

1. **守护进程**
    1. **定义**：守护进程（Daemon 进程）是在后台长期运行的服务进程，它独立于控制终端，没有交互界面。这类进程主要用于提供系统级服务，像 Web 服务器 `nginx`、数据库服务 `mysql` 等都属于守护进程。它们不受用户登录或注销操作的影响，一般在开机时自动启动，直至系统关闭才终止运行。
    2. **实现思路**：创建标准的守护进程，需要借助 `fork` 函数，并配合一系列特定操作，以此确保进程能够脱离终端和父进程的控制。具体核心步骤如下：
        - **第一次 fork**：父进程通过 `fork` 创建子进程后，立即退出。这使得子进程成为“孤儿进程”，随后由 init 进程（在现代系统中，通常是 systemd 等初始化进程）接管，从而脱离原终端的控制。
        - **设置会话 ID（setsid）**：子进程调用 `setsid()` 函数，创建一个新的会话，并成为该会话的组长。这一操作能让进程彻底摆脱原终端所属的进程组。
        - **第二次 fork（可选）**：再次通过 `fork` 创建子进程，然后当前进程退出。此步骤可避免子进程成为会话组长，防止其意外获取终端。
        - **切换工作目录**：通常将工作目录切换到根目录（使用 `chdir("/")`）。这样做可避免进程占用可卸载的文件系统，确保进程运行不受文件系统挂载状态的干扰。
        - **关闭文件描述符**：关闭从父进程继承而来的与终端相关的标准输入（文件描述符 0）、标准输出（文件描述符 1）和标准错误（文件描述符 2）等文件描述符，防止进程与原终端产生不必要的交互。
        - **重定向文件描述符**：将标准输出和标准错误重定向到 `/dev/null` 或日志文件。这样既可以避免进程输出无意义的内容，又便于记录进程运行过程中的重要信息。
    3. **Linux 系统中的查看方式**：在 Linux 系统中，守护进程通常在命名上以 `d` 结尾（如 `sshd`、`crond`）。可通过以下命令查看：
        - `ps -ef | grep -v grep | grep -E 'daemon|d$'`：该命令用于筛选进程名称中包含 `daemon` 或结尾为 `d` 的进程，但这并非绝对的判断标准，仅作参考。
        - `ps aux | grep '^root'`：由于多数守护进程以 root 身份运行，结合此条件可更精准地筛选守护进程。
        - `systemctl list-units --type=service`：此命令用于查看系统服务，因为多数系统服务对应的进程就是守护进程。

2. **后台进程**
    1. **定义**：后台进程指在终端中启动后，不会阻塞终端，用户能够在终端继续执行其他命令的进程。一般通过在命令末尾加上 `&` 符号来启动后台进程。
    2. **与守护进程的区别**：后台进程虽也在后台运行，但与守护进程存在明显差异。后台进程通常仍与终端存在某种联系，例如其输出可能会显示在终端上，当终端关闭时，后台进程可能会受到影响（除非采取特殊处理）。而守护进程则完全脱离终端，独立运行，不受终端状态的影响。在功能和应用场景方面，后台进程常用于执行一些相对独立且不需要用户实时关注的短期任务。例如，在终端中启动一个数据处理脚本，在脚本执行过程中，用户仍可使用终端进行其他操作。守护进程则更多用于提供持续的系统级服务，为整个系统或网络中的其他组件提供支持，如 Web 服务器守护进程持续监听网络请求，为客户端提供网页服务。

简单概括，守护进程可视为 “系统的后台服务”，后台进程则是 “用户的后台任务”。 

##### 1.1.2 孤儿进程和僵尸进程

1. **孤儿进程**
    - **定义**：当一个父进程退出，而它的一个或多个子进程仍在运行时，这些子进程便成为孤儿进程。孤儿进程会被 `init` 进程（进程号为 1）收养，由 `init` 进程负责完成它们的状态收集工作。
    - **特点**：孤儿进程不会对系统造成危害，因为 `init` 进程能够妥善处理它们的状态收集与资源释放。

2. **僵尸进程**
    - **定义**：一个进程使用 `fork` 创建子进程后，如果子进程退出，而父进程没有调用 `wait` 或 `waitpid` 获取子进程的状态信息，子进程的进程描述符仍会保存在系统中，这类进程即为僵尸进程。在 Linux 系统里，僵尸进程是已退出但资源未被回收的进程。子进程退出时，内核会保留其 PID、退出状态等信息，等待父进程通过 `wait()` 或 `waitpid()` 回收；若父进程未调用这些函数，子进程就会成为僵尸进程，其状态标记为 `Z`（Zombie）。
    - **产生原因**
        - 父进程未调用 `wait`/`waitpid` 系统调用回收子进程资源。
        - 父进程存在逻辑缺陷，例如忽略子进程退出信号 `SIGCHLD`，或者长期运行但未处理子进程状态。
    - **查看方法（Linux 系统）**
        - 使用 `ps` 命令查看状态为 `Z` 的进程：
        ```bash
        ps -ef | grep Z  # 状态列显示 Z 的即为僵尸进程
        # 或更直观的格式：
        ps aux | awk '$8=="Z"'  # 筛选状态（第8列）为 Z 的进程
        ```
    - **特点**
        - 任何一个子进程（`init` 除外）在 `exit()` 之后，不会立即消失，而是留下一个称为僵尸进程（Zombie）的数据结构，等待父进程处理。
        - 僵尸进程的进程号会一直被占用，而系统所能使用的进程号有限。若大量产生僵尸进程，会因进程号耗尽导致系统无法产生新进程。
    - **危害**：僵尸进程占用系统的进程号资源，若不及时处理，可能致使系统无法创建新进程。
    - **解决办法**
        - **父进程主动回收**：父进程通过 `wait()` 或 `waitpid()` 系统调用获取子进程退出状态，内核随即释放僵尸进程的资源。示例代码如下：
        ```c
        #include <sys/wait.h>
        int status;
        wait(&status);  // 阻塞等待任意子进程退出并回收
        ```
            - **父进程处理 `SIGCHLD` 信号**：子进程退出时会向父进程发送 `SIGCHLD` 信号，父进程可注册信号处理函数，在信号触发时调用 `wait` 回收。示例代码如下：
        ```c
        #include <signal.h>
        void handle_sigchld(int sig) {
            wait(NULL);  // 收到信号后回收子进程
        }
        signal(SIGCHLD, handle_sigchld);  // 注册信号处理
        ```
        - **终止父进程**：若父进程无法修复，终止父进程后，僵尸进程会被 `init` 进程（或 `systemd`）接管，`init` 会自动调用 `wait` 回收其资源。

    - **僵尸进程的危害场景**
        * 场景描述：假设有一个进程定期产生子进程，这些子进程执行少量任务后即退出，生命周期很短。然而，父进程只管生成新子进程，对子进程退出后的情况不管不顾。一段时间后，系统中就会出现大量僵尸进程，使用 `ps` 命令查看，会看到许多状态为 `Z` 的进程。
        * 根本原因：严格来讲，僵尸进程并非问题根源，产生大量僵尸进程的父进程才是罪魁祸首。
        * 处理方法：终止产生大量僵尸进程的父进程（通过 `kill` 发送 `SIGTERM` 或者 `SIGKILL` 信号）。之后，这些僵尸进程会变成孤儿进程，被 `init` 进程接管。`init` 进程会调用 `wait()` 释放它们占用系统进程表中的资源，从而清理掉这些僵死的孤儿进程。 

#### 1.2 进程 ID

进程通过以下 ID 实现系统级管理和区分：

| 标识     | 全称      | 作用                                                                        |
| -------- | --------- | --------------------------------------------------------------------------- |
| **PID**  | 进程 ID   | 唯一标识一个进程（如 `ps` 命令显示的第一列数字）。                          |
| **PGID** | 进程组 ID | 多个相关进程组成进程组（如管道中的多个命令），PGID 通常等于组长进程的 PID。 |
| **SID**  | 会话 ID   | 一个会话包含多个进程组（如终端启动的所有进程），用于管理终端相关进程。      |

**示例**：在终端运行 `ls | grep txt` 时，`ls` 和 `grep` 属于同一进程组（共享 PGID），且同属终端启动的会话（共享 SID）。

#### 1.3 进程的组成

一个进程由 4 部分核心组件构成，它们对于进程的运行和管理缺一不可：

1. **进程控制块（PCB，Process Control Block）** PCB 如同进程的“身份证”，是内核维护的结构体，记录着进程所有关键信息，也是进程存在的唯一标志（当 PCB 被释放时，进程即消亡）。不同操作系统中 PCB 的具体实现可能略有不同，其通常包含以下几类信息： 

  - **进程描述信息**：进程标识符（PID）、进程组标识符（PGID）、会话标识符（SID）以及父进程标识符（PPID）等，用于唯一标识进程及其所属关系。 
  - **进程状态信息**：包括运行态（R）、就绪态（S）、阻塞态（D）、僵尸态（Z）等进程状态，还有进程优先级、未决信号等信息，这些决定了进程在系统中的调度和执行顺序。 
  - **资源分配清单**：进程打开的文件描述符（如标准输入 `stdin`、标准输出 `stdout` 等）、内存页表（负责将虚拟地址映射到物理地址，实现进程对内存资源的管理）以及信号量等，记录了进程运行过程中所占用的各类资源。 
  - **CPU 上下文**：包含寄存器（如程序计数器 PC，用于指示下一条要执行的指令地址；栈指针 SP，指向当前进程栈的栈顶）、状态寄存器（记录 CPU 当前的状态）等信息。这些信息在进程切换时发挥关键作用，能够确保进程在重新执行时恢复到之前的运行状态。

2. **数据段**：进程运行过程中使用的各种数据，例如程序中定义的变量等，都存储在此段。它是进程处理和操作数据的存储区域。
3. **程序段**：即程序的代码，是由一系列指令序列构成的，规定了进程要执行的具体操作和任务逻辑。
4. **堆栈段（文本未提及，但通常是进程组成部分）**：用于存储函数调用的参数、局部变量以及返回地址等。栈随着函数调用和返回进行动态的生长和收缩，为进程的函数调用和执行提供必要的环境支持。

进程控制块是操作系统对进程进行管理的重要依据，几乎所有与进程管理相关的操作都依赖于 PCB 中的内容。例如，进程调度程序依据 PCB 中的进程状态和优先级信息来决定哪个进程获得 CPU 资源；内存管理模块通过 PCB 中的内存页表信息来管理进程的内存使用等。

#### 1.4 进程切换（Context Switch）

**进程切换**是指 CPU 从一个进程切换到另一个进程的过程，通常由内核调度器完成，触发条件主要包括：

* **1. 主动让出 CPU**

    * **阻塞操作**：进程执行 `sleep`、`wait` 等阻塞调用（如等待 I/O 或信号）时，进入阻塞态，自动放弃 CPU。
    * **主动让出（yield）**：进程调用 `yield` 表示愿意让出 CPU，调度器可选择是否响应。

* **2. 被动剥夺 CPU**

    * **时间片耗尽**：在分时系统中，进程用完分配的时间片后，调度器强制进行切换。
    * **优先级抢占**：高优先级进程进入就绪状态时，可抢占当前运行的低优先级进程。
    * **硬件中断触发**：如 I/O 完成产生中断，唤醒阻塞进程后可能引发调度。

* **3. 进程切换步骤**

    1. **保存当前进程上下文**：将当前进程的寄存器值（程序计数器 PC、栈指针 SP、通用寄存器等）及状态保存至其 PCB（进程控制块）中。
    2. **更新 PCB 信息**：将当前进程状态从“运行”更新为“就绪”或“阻塞”，并调整其在对应队列中的位置。
    3. **选择下一个进程**：调度器根据调度算法（如时间片轮转、优先级）从就绪队列中选出下一个运行的进程。
    4. **恢复目标进程上下文**：从选中进程的 PCB 中读取保存的寄存器数据，恢复执行状态。
    5. **切换地址空间**：更新内存管理单元（MMU）的页表寄存器，切换至目标进程的地址空间，确保内存访问的隔离与正确性。

    **示例**：当进程 A 运行到 `sleep(1)` 时主动阻塞，此时就会触发上述 CPU 从进程 A 到进程 B 的切换流程。1 秒后进程 A 被唤醒，重新进入就绪队列，等待再次被调度。当再次被调度时，进程 A 将从 `sleep` 之后的指令继续恢复执行。

#### 1.5 进程调度算法

| 分类     | 特点         | 是否抢占 | 优劣分析           | 调度原理简介                           |
| ------ | ---------- | ---- | -------------- | -------------------------------- |
| FCFS   | 到达顺序调度     | ✖    | 实现简单；长作业占据资源   | 按进程到达顺序排队，先到先运行，直到完成。            |
| SJF    | 优先最短作业     | ✖    | 平均等待时间低；易饿死长作业 | 每次选择预计运行时间最短的进程执行，不可抢占。          |
| SRTF   | 剩余时间最短优先   | ✔    | 响应快；频繁切换开销大    | 当前进程运行时，若有新进程剩余时间更短则抢占执行。        |
| HRRF   | 响应比最高优先    | ✖    | 平衡等待时间；计算复杂    | 每次计算所有作业的响应比，选择响应比最高的执行。         |
| 时间片轮转  | 均分时间片轮流执行  | ✔    | 公平性好；响应及时      | 所有进程按顺序轮流执行，每次最多执行一个时间片。         |
| 多级反馈队列 | 多级优先级与动态调整 | ✔    | 综合适用性强，复杂度高    | 新进程从高优先级队列开始，未完成则降级，结合时间片和优先级调度。 |

### 2. 进程创建

### 2. 进程创建机制详解

进程创建是操作系统中最核心的功能之一。在类 Unix 系统中，进程创建通常通过 `fork()`、`vfork()`、`clone()` 等系统调用完成，其中 `exec()` 系列函数常与之配合使用，用于在新创建的进程中加载和执行不同的程序。

#### 2.1 进程创建的一般流程

1. **复制父进程资源（通常由 `fork()` 触发）**
   当父进程调用 `fork()` 时，内核会创建一个新的子进程，并复制父进程的关键资源，包括地址空间（代码段、数据段、堆栈段）、打开的文件描述符、信号处理方式等。为了提高效率，大多数现代操作系统采用**写时复制**（Copy-On-Write, COW）机制。该机制下，父子进程最初共享相同的物理内存页，直到某一方尝试修改共享内存时，内核才会真正进行内存页的复制。

2. **分配唯一的进程标识符（PID）**
   子进程创建完成后，内核会为其分配一个唯一的 PID（Process Identifier），并将其状态设置为“就绪”（Ready），准备参与调度。

3. **执行新程序（可选，`exec` 系列函数）**
   子进程创建后，若希望运行一个不同的程序，可以调用 `exec` 系列函数（如 `execl`、`execv`、`execve` 等），将当前进程映像完全替换为新程序的映像。此过程不会创建新进程，PID 不变。
   例如：

   ```cpp
   execl("/bin/ls", "ls", "-l", NULL);
   ```

   上述代码会用 `/bin/ls` 程序替换当前进程，传入参数 `-l`。

4. **进入调度队列**
   创建与初始化完成后，子进程被加入调度器的就绪队列，等待根据调度策略分配 CPU，进入运行状态。

#### 2.2 `fork()`、`vfork()` 与 `clone()` 的区别与使用

| 函数名       | 是否共享地址空间 | 是否立即执行子进程 | 用途与特点                                            |
| --------- | -------- | --------- | ------------------------------------------------ |
| `fork()`  | 否（写时复制）  | 是         | 标准进程创建方式，父子进程资源相互独立，安全性高，但开销较大                   |
| `vfork()` | 是（完全共享）  | 是（父进程阻塞）  | 高效替代 `fork`，常用于创建后立刻执行 `exec` 的子进程；使用不当易导致数据破坏   |
| `clone()` | 可选（参数控制） | 是         | Linux 特有的系统调用，提供精细化控制，可用于线程创建或构建轻量级进程，支持指定资源共享粒度 |

##### 2.2.1 `fork()`

* 用法示例：

  ```cpp
  pid_t pid = fork();
  if (pid == 0) {
      // 子进程代码
  } else {
      // 父进程代码
  }
  ```

* 特点：创建的子进程与父进程几乎完全相同，仅 PID 和返回值不同。采用写时复制机制，适用于需要完整资源隔离的场景。

##### 2.2.1 `vfork()`

* 用法与 `fork()` 相同，但执行语义不同：

  ```cpp
  pid_t pid = vfork();
  if (pid == 0) {
      execl("/bin/ls", "ls", "-l", NULL); // 推荐立即执行 exec
  }
  ```

* 特点：父子进程共享地址空间，子进程运行期间父进程会被挂起。效率更高，但不允许子进程在 `exec` 前修改数据，否则会影响父进程。适合“创建即 exec”的场景。

##### 2.2.1 `clone()`

* 是 Linux 提供的底层系统调用，其函数原型为：

  ```c
  int clone(int (*fn)(void *), void *child_stack, int flags, void *arg, ...);
  ```

* 通过 `flags` 参数可控制资源共享行为（如共享文件描述符、地址空间、信号处理器等），是实现线程库（如 pthread）和容器技术（如 LXC、Docker）的核心机制。

* 示例（模拟线程创建）：

  ```c
  clone(thread_func, child_stack, CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND, arg);
  ```

#### 2.3 `exec` 系列函数详解

`exec` 系列函数用于用新的程序替换当前进程映像。它们不创建新进程，而是在当前进程上下文中加载新程序并开始执行。调用成功后，原进程代码不会再继续执行；调用失败才会返回。

常见函数及区别如下：

| 函数名      | 参数形式                  | 是否可传环境变量 | 说明                     |
| -------- | --------------------- | -------- | ---------------------- |
| `execl`  | 可变参数（arg0, ..., NULL） | ✖        | 简洁直观，适用于参数较少时          |
| `execv`  | 参数数组 `argv[]`         | ✖        | 可动态构造参数列表，适用于参数较多或不确定时 |
| `execle` | 可变参数 + `envp[]`       | ✔        | 支持传入自定义环境变量            |
| `execve` | `argv[]` + `envp[]`   | ✔        | 系统调用底层实现形式，其他变种最终调用它   |

#### 3.1 管道和FIFO

管道是 UNIX 系统中最早的进程间通信（IPC）机制之一，属于**半双工**通信方式，即数据只能在一个方向上流动。其本质是一个**内核缓冲区**，数据以**先进先出**（FIFO）方式传递。一端进程顺序写入数据，另一端顺序读取。该缓冲区可以看作一个循环队列，读写位置自动推进，**数据只能被读取一次**，读出后即从缓冲区中移除。

当缓冲区**写满**或**读空**时，写入或读取进程会被阻塞，并加入等待队列；一旦有空位或新数据写入，系统会唤醒相应的进程继续操作。

管道主要分为两种：

* **匿名管道（无名管道）**：由 `pipe()` 创建，仅允许**具有亲缘关系的进程**（如父子或兄弟进程）之间通信。其创建方式如下：

  ```c
  #include <unistd.h>
  int pipe(int fd[2]); // fd[0] 读端，fd[1] 写端；成功返回0，失败返回-1
  ```

  匿名管道只存在于内存中，不属于任何文件系统。读写操作可通过 `read()`、`write()` 等标准文件操作函数完成。

    <img src="https://cs-notes-1256109796.cos.ap-guangzhou.myqcloud.com/53cd9ade-b0a6-4399-b4de-7f1fbd06cdfb.png"/>
  </div><br>

* **命名管道（FIFO）**：去除了匿名管道只能用于亲缘进程间通信的限制，可在**无关进程之间**实现数据传输。FIFO 是一种存在于文件系统中的特殊类型文件，具备管道的通信特性，也支持使用标准文件 I/O 接口进行读写。数据同样采用“先进先出”方式传输，读取后即从管道中移除。

  FIFO 通过 `mkfifo()` 创建：

  ```c
  #include <sys/stat.h>
  int mkfifo(const char *pathname, mode_t mode); // 成功返回0，失败返回-1
  ```

  `mode` 参数与 `open()` 中一致，指定文件权限。

  打开 FIFO 时的阻塞行为与是否指定 `O_NONBLOCK` 标志相关：

  * 默认阻塞：若不指定 `O_NONBLOCK`，只读打开会阻塞直到有写端打开，反之亦然；
  * 非阻塞：若指定 `O_NONBLOCK`，只读打开会立即返回；写端若无进程打开读端，则失败返回，`errno` 被置为 `ENXIO`。

  FIFO 常用于**客户端-服务器模型**，作为数据交换的汇聚通道。

#### 3.2 消息队列

消息队列是一个消息的链表，是一系列保存在内核中的消息列表。用户进程可以向消息队列添加消息，也可以从消息队列读取消息。与管道通信相比，消息队列的优势在于：每条消息可以指定特定的消息类型，接收时不需要严格按照队列次序，可以根据自定义条件选择读取特定类型的消息。

可以把消息看作一个具有特定格式和优先级的记录。具有写权限的进程可以按照一定规则向消息队列中添加消息，具有读权限的进程可以读取消息队列中的消息。

* 消息队列是**消息的链表，存放在内核中**，由一个标识符（队列 ID）标识。
* 消息队列是**面向记录**的（消息是一条一条的），每条消息具有格式和优先级。
* 消息队列**独立于发送和接收进程**，即使进程终止，消息队列本身及其内容仍然存在，除非显式删除。
* 消息队列可以实现消息的**随机读取**，可以按类型读取消息，而不是只能按照先进先出（FIFO）顺序。

消息队列的基本接口如下：

```c
#include <sys/msg.h>

// 创建或打开消息队列，成功返回队列ID，失败返回 -1
int msgget(key_t key, int msgflg);

// 向消息队列发送消息，成功返回 0，失败返回 -1
int msgsnd(int msqid, const void *msgp, size_t msgsz, int msgflg);

// 从消息队列接收消息，成功返回消息数据的长度，失败返回 -1
int msgrcv(int msqid, void *msgp, size_t msgsz, long msgtyp, int msgflg);

// 控制消息队列（如删除队列），成功返回 0，失败返回 -1
int msgctl(int msqid, int cmd, struct msqid_ds *buf);
```

关于 `msgget` 函数的行为：

* 如果没有对应 `key` 的消息队列，并且 `msgflg` 包含 `IPC_CREAT` 标志，则创建一个新队列；
* 如果 `key` 为 `IPC_PRIVATE`，则总是创建新的队列。

使用 `msgrcv` 接收消息时，`msgtyp` 参数的含义如下：

* `msgtyp == 0`：读取队列中**第一条消息**（即FIFO）；
* `msgtyp > 0`：读取队列中类型等于 `msgtyp` 的**第一条消息**；
* `msgtyp < 0`：读取队列中消息类型值小于等于 `|msgtyp|` 的所有消息中**类型值最小的一条**。

这使得消息队列支持根据“类型”进行非顺序读取，可以将 `msgtyp` 看作一种优先级权值。

相比于管道（尤其是 FIFO），消息队列具有以下优点：

* **生命周期独立**：消息队列可以在进程退出后继续存在，避免了管道打开/关闭时的同步问题；
* **解耦性强**：发送方和接收方不需要同时存在，也不需要同步阻塞；
* **支持选择性读取**：可以通过消息类型读取感兴趣的消息，而管道只能顺序读取；
* **内存速度更快**：消息队列存在于内核内存，而 FIFO 是基于文件的机制，通常涉及磁盘，访问速度慢；
* **数据块式传输**：每次读写是一个完整的消息单元；而管道是字节流式，不具备结构和边界。

另外，匿名管道是进程间的短生命周期通信手段，依附于进程；而消息队列由内核维护，生命周期更长，可跨多个进程、跨父子关系使用。

示例函数原型：

```c
#include <sys/types.h>
#include <sys/ipc.h>
#include <sys/msg.h>

int msgget(key_t key, int msgflg);
```

总结：消息队列是一种功能更强、使用更灵活的进程间通信机制，特别适合用于**异步通信、多进程协作、有优先级或分类需求的消息传递场景**。

#### 3.3 信号(signal)

信号（Signal）是 Linux 系统中一种**用于进程间通信或异步事件通知的机制**，本质上是软件层面对中断机制的模拟，允许内核或其他进程**异步地通知某个进程某个事件的发生**。

* 信号是 Linux 中一种重要的、**异步的进程间通信机制**；
* 可由硬件触发、软件调用或异常产生；
* 提供三种处理机制：**默认处理、捕捉处理、自行忽略**；
* 某些信号不可被阻塞或忽略（如 `SIGKILL`、`SIGSTOP`）；
* 信号处理过程涉及**进程状态切换、堆栈保护与恢复、程序计数器（PC）重定向**等底层核心机制，是操作系统调度和异常处理的重要组成部分。

* **信号的基本特性**

  * 信号可以在**任意时间**发送给进程，不受进程当前状态限制；
  * 如果进程**暂未运行**，信号会被内核**暂存为未决信号**，待合适时机再传递；
  * 若进程对某信号进行**阻塞处理**，则信号传递将被**延迟**；
  * 信号通信开销小，适合处理**异步事件通知**，但其传递机制较为复杂；
  * 使用 `<signal.h>` 头文件中定义的 API（如 `signal()`、`sigaction()`）进行信号操作和处理。

* **信号的来源**

信号可以由以下三类事件产生：

1. **硬件来源**
   例如用户在终端按下 `Ctrl+C`，终端驱动程序会触发 `SIGINT` 信号发送给前台进程。

2. **软件来源**
   主要包括以下系统调用：

   * `kill(pid, sig)`：向指定进程或进程组发送信号；
   * `raise(sig)`：向当前进程自身发送信号；
   * `sigqueue(pid, sig, value)`：向进程发送带有参数的信号（实时信号）；
   * `setitimer()` / `alarm()`：设置定时器，到期后触发 `SIGALRM`；
   * `abort()`：异常终止进程并发送 `SIGABRT`；
   * 程序运行中出现严重错误，如非法内存访问（段错误）、除以零、栈溢出等，也会触发对应信号。

   💡 某些信号如 `SIGSEGV`、`SIGABRT` 会自动生成 core dump 文件用于调试；但 `SIGKILL` 和 `SIGINT` 不会。

3. **命令行工具与终端操作**

   * 使用 `kill` 命令可以手动向进程发送信号；
   * 键盘快捷键：

     * `Ctrl+C` 触发 `SIGINT`；
     * `Ctrl+Z` 触发 `SIGTSTP`；
     * `Ctrl+\` 触发 `SIGQUIT`（并产生 core dump）。

* **进程对信号的响应方式**

进程收到信号后，Linux 提供三种处理方式：

1. **默认处理**
   每种信号都有默认处理行为，如终止、忽略、暂停等。例如 `SIGSEGV` 默认终止进程并生成 core dump。

2. **捕捉信号**
   用户可以通过注册信号处理函数（如 `signal()` 或 `sigaction()`）自定义信号的处理逻辑。

3. **忽略信号**
   某些非关键信号可以被设置为忽略，使其不影响进程运行。

⚠️ 注意：以下两个信号无法被捕捉、忽略或阻塞：

* `SIGKILL (9)`：无条件终止进程；
* `SIGSTOP (19)`：无条件挂起进程。

* **信号的处理流程**

1. **信号接收**
   当内核接收到信号时，会将其记录到进程控制块（PCB）的**未决信号集**中。此时，信号尚未真正“被处理”，进程对其无感知。内核会通知进程中断，促使其从用户态陷入内核态，准备处理信号。

2. **信号检测**
   信号检测不会立即发生，而是延迟到：

   * 进程**从内核态返回用户态**前；
   * 或进程**从休眠状态被唤醒**时。

   若检测到未决信号，则进入信号处理阶段。

3. **信号处理**

   * 若该信号未注册处理函数，则执行**默认操作**；
   * 若已注册处理函数，则：

     * 内核将当前内核栈保存到用户栈；
     * 修改程序计数器（PC），指向信号处理函数；
     * 回到用户态，执行用户定义的处理函数；
     * 处理函数执行完毕后再次陷入内核；
     * 恢复之前的栈和程序计数器，继续执行原用户程序。

4. **多个信号的处理顺序**
   若有多个信号待处理，则内核按优先级和到达顺序依次处理它们。

* **信号的处理时机**

信号具有**延迟处理**特性：

* 即使信号已经被送达进程，也**不会立即生效**；
* 只有当进程**从内核态返回用户态**或**从睡眠中被唤醒**时，内核才会检查并处理未决信号；
* 因此，信号并非实时触发，而是**中断+延迟处理**的混合机制。

* **常见信号表（部分）**

| 信号名       | 编号 | 含义           |
| --------- | -- | ------------ |
| `SIGHUP`  | 1  | 终端挂断         |
| `SIGINT`  | 2  | 中断（如 Ctrl+C） |
| `SIGKILL` | 9  | 强制终止进程，不能捕捉  |
| `SIGTERM` | 15 | 请求程序正常终止     |
| `SIGSTOP` | 19 | 强制暂停进程，不能捕捉  |
| `SIGSEGV` | 11 | 段错误，非法内存访问   |
| `SIGALRM` | 14 | 定时器超时信号      |
| `SIGUSR1` | 10 | 用户自定义信号 1    |
| `SIGUSR2` | 12 | 用户自定义信号 2    |

信号机制在 Linux 系统中应用广泛，既可用于进程间的简单通信，也可用于实现定时、异常捕捉、中断响应等功能。虽然功能强大，但由于其异步性和系统层级较高的复杂性，在使用时需要谨慎设计，确保信号处理函数的可重入性和简洁性，避免竞态条件或死锁等问题。

#### 3.4 信号量

 
信号量（Semaphore）是一种特殊的整型变量，主要用于实现**进程或线程之间的互斥与同步**，而非用于数据通信。与其他进程间通信（IPC）机制如管道、消息队列或共享内存不同，信号量不传递数据，其核心作用是**协调多个执行单元对共享资源的访问顺序**，以避免竞争条件或死锁。

信号量操作依赖操作系统提供的底层支持，所有操作均具有**原子性**，可确保在并发环境中不会发生竞态问题。其基本原语包括等待（P，Proberen）和发送（V，Verhogen）操作：

* `P(sv)`：尝试将信号量值减 1。若信号量值大于 0，立即减 1 并继续执行；否则当前进程被挂起，直到信号量值大于 0。
* `V(sv)`：将信号量值加 1，若有进程因等待此信号量而挂起，则唤醒其中一个进程。

信号量支持**对单个或多个资源单位进行计数**，因此在实际应用中，不仅可以控制互斥访问（将初始值设为 1），也可用于同步多个生产者/消费者或任务调度场景。此外，信号量也可支持“信号量组”概念，即一次对多个信号量进行批量操作。


信号量是解决进程间同步和互斥问题的重要工具。其典型应用包括：

* **互斥锁（Mutex）**：使用初始值为 1 的信号量，确保同一时刻仅有一个进程访问关键区；
* **资源计数器**：通过初始值设定总资源数，进程按需申请/释放，防止资源争用；
* **生产者-消费者模型**：配合共享内存使用，一个信号量控制缓冲区空位，另一个控制已有数据量，实现生产与消费的有序交替；

信号量机制底层依赖内核调度器进行挂起与唤醒处理，其操作开销虽小于进程间通信，但高于线程间互斥工具（如 mutex）。在性能要求较高或资源频繁访问的场景中，可结合共享内存、消息队列等机制提高效率。

信号量的使用虽然灵活强大，但也容易引入死锁等问题，因此在编程时需特别注意对称使用（每个 P 操作应有对应的 V 操作），并设定合理的操作顺序和资源申请策略。

如需进一步结合实例展示典型使用场景，可继续补充生产者-消费者模型的代码示例及分析。


Linux 提供了 System V 信号量的接口，核心函数包括：

```c
#include <sys/sem.h>
```

##### （1）`semget`

```c
int semget(key_t key, int num_sems, int sem_flags);
```

* 创建或获取一个信号量集合；
* `key`：信号量集合的标识符；
* `num_sems`：集合中信号量的数量（新建时有效，通常设为 1）；
* `sem_flags`：权限标志位，常与 `IPC_CREAT` 联合使用；
* 返回：成功返回信号量集 ID，失败返回 -1。

##### （2）`semop`

```c
int semop(int semid, struct sembuf semoparray[], size_t numops);
```

* 对一个或多个信号量进行操作；
* 所有操作均为原子操作，执行过程中不会被中断；
* 返回：成功返回 0，失败返回 -1，并设置 `errno`。

`sembuf` 结构定义如下：

```c
struct sembuf {
    unsigned short sem_num; // 信号量在集合中的编号，范围为 0 ～ num_sems - 1
    short sem_op;           // 操作值：正值、负值或 0
    short sem_flg;          // 操作标志位，可取 IPC_NOWAIT 或 SEM_UNDO
};
```

`sem_op` 的含义如下：

* **`sem_op > 0`**：表示释放资源，将该值加到信号量上；若有进程等待该信号量，将唤醒等待队列中的进程；
* **`sem_op < 0`**：表示请求资源。若当前信号量值足够（≥ 绝对值），立即减去该值；否则：

  * 若设置了 `IPC_NOWAIT`，立即失败，返回 `EAGAIN`；
  * 否则进程进入睡眠状态，挂起在该信号量的等待队列中，直到满足资源、被信号打断或信号量被删除；
* **`sem_op == 0`**：表示等待信号量值为 0，常用于同步场景；

  * 若当前值为 0，立即返回；
  * 若不为 0，行为同上，依赖 `sem_flg` 决定是否阻塞。

等待中被打断时，`semop` 可能返回：

* `EINTR`：等待期间接收到信号；
* `EIDRM`：等待期间信号量被删除；
* `EAGAIN`：未设置阻塞，资源不足。

##### 2. `semctl` 控制信号量属性

```c
int semctl(int semid, int sem_num, int cmd, ...);
```

* 可用于设置初始值、读取状态、删除信号量等；
* 常见命令包括：

###### （1）`SETVAL`

用于初始化指定编号的信号量。调用时需传入一个 `union semun` 类型的参数：

```c
union semun {
    int val;                // 用于 SETVAL
    struct semid_ds *buf;   // 用于 IPC_STAT, IPC_SET
    unsigned short *array;  // 用于 GETALL, SETALL
    struct seminfo *__buf;  // 用于 IPC_INFO (Linux-specific)
};
```

例如，设置信号量初值为 1，用于互斥访问：

```c
union semun arg;
arg.val = 1;
semctl(semid, 0, SETVAL, arg);
```

###### （2）`IPC_RMID`

用于从系统中移除信号量集合。信号量不会自动销毁，程序退出后仍会保留在内核中，必须显式调用 `IPC_RMID` 释放资源，否则可能引发资源泄露或程序异常。


#### 3.5 共享内存

共享内存是最快的一种进程间通信（IPC）方式，它允许多个进程直接访问同一块内存区域。由于数据不需要在内核和用户空间之间来回拷贝，因此共享内存的通信效率远高于消息队列、管道或套接字。

* **共享内存是在内核中开辟一块区域，不同进程通过映射访问同一块物理内存**

* 共享内存由一个标识符（即共享内存 ID）来标识，**创建后并不会随着进程退出而自动删除**

* 原型

```c
#include <sys/ipc.h>
#include <sys/shm.h>

// 创建或获取共享内存：成功返回共享内存ID，失败返回-1
int shmget(key_t key, size_t size, int shmflg);

// 映射共享内存：返回映射后的地址，失败返回(void *) -1
void *shmat(int shmid, const void *shmaddr, int shmflg);

// 解除映射：成功返回0，失败返回-1
int shmdt(const void *shmaddr);

// 控制共享内存：成功返回0，失败返回-1
int shmctl(int shmid, int cmd, struct shmid_ds *buf);
```

使用流程：

1. 调用 `shmget` 创建或获取共享内存段。
2. 使用 `shmat` 将共享内存段映射到进程地址空间。
3. 读写共享内存中的数据（像普通指针一样操作）。
4. 使用 `shmdt` 解除映射。
5. 使用 `shmctl` 来删除共享内存段或设置属性。

示例代码：

```c
// 创建共享内存
int shmid = shmget(IPC_PRIVATE, 1024, IPC_CREAT | 0666);
if (shmid == -1) { perror("shmget"); exit(1); }

// 映射到进程地址空间
char *p = (char *)shmat(shmid, NULL, 0);
if (p == (char *)-1) { perror("shmat"); exit(1); }

// 写入数据
strcpy(p, "Hello Shared Memory");

// 解除映射
shmdt(p);

// 删除共享内存（通常由父进程或最后一个进程调用）
shmctl(shmid, IPC_RMID, NULL);
```

与其他通信方式的比较：

* 相比消息队列：

  * 消息队列适合传递结构化、少量数据；
  * 共享内存适合传递大批量数据（如图像、数组），效率更高；
  * 消息队列由内核维护队列结构，而共享内存由用户自己同步（需使用信号量或互斥锁保证并发访问安全）；
* 相比管道和 FIFO：

  * 共享内存具有更高的传输速度；
  * 管道是单向、字节流式通信，而共享内存是双向、块式直接访问；
* 相比套接字：

  * 套接字可以跨主机通信，但性能不如共享内存。

总结：

共享内存是最快的 IPC 机制，适用于进程间高频、大量数据的通信场景，但它要求开发者自己负责访问同步控制，通常搭配信号量等机制使用。
